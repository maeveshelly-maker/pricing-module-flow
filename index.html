Carrier_Association.Preferred_Price_Book__c lookup field · PricebookEntry.Floor__c / Ceiling__c custom fields · a validation rule that hard-blocks the save if the rep's chosen Pricebook doesn't match the carrier's Preferred_Price_Book__c (or Standard if none) · the Proposal/RFP stage-gate validation rule · Slack-routed approval for out-of-range pricing — likely reusing the contractor's existing Slack approval object rather than native Approval Process or a new custom object, pending confirmation it's generic enough to extend.

No Flow sets the Pricebook — this is a manual rep action, blocked if wrong. The rep clicks Add Product on the Opportunity, and that's the moment Salesforce natively prompts them to choose a Pricebook before they can pick a specific product. Preferred_Price_Book__c is a reference field, not automation — it doesn't set anything by itself, but the validation rule checks against it and hard-blocks the save if the rep picks wrong. Example: if the carrier is Accelerant (which has preferred pricing), the rep cannot save with the Standard Pricebook selected — the rule blocks it and forces them to pick Accelerant's book instead.

Changing a floor/ceiling or negotiated rate: edit access restricted to Nick/Martha via a permission set, with Field History Tracking on — tracked, not approval-gated. Changes aren't retroactive: a deal already quoted keeps the range it was checked against at the time; the live PricebookEntry is always "current," not versioned per deal.

Flow vs. validation rule: a validation rule can't set a value — it can only block or allow a save. So it can't pick the Pricebook for the rep; it can only catch it afterward if they picked the wrong one for this carrier. Checking whether the entered price is within Floor__c/Ceiling__c is the same kind of check — a pure block/allow. Either way, the actual rate + floor + ceiling for a carrier still has to be entered by Nick/Martha — the validation rule enforces against that data, it doesn't replace the need to build it.

Example — anchor per LOB: an Auto-only deal requires one anchor: Auto Bodily Injury. An Auto + GL deal requires two: Auto Bodily Injury and GL's equivalent primary line. Litigation, Glass, Towing, Cargo, Trailer/Interchange, etc. are never required to advance, regardless of how many LOBs are in play.

Why anchor-only at the gate: this isn't just about keeping the gate light — it's what gets a real dollar amount into Salesforce as early as Proposal/RFP, instead of waiting until a contract is signed to know what a program is worth. That's the original problem this project solves: revenue estimates for new programs were guesses. One required anchor line fixes that immediately, before the rest of the exhibit even exists.

Full catalog loads now — the gate is a separate decision from the catalog. The pricing repository sheet already has typical/min/max/median for all 29 coverage lines — that analysis is already done, so there's no reason to hold the rest back from Salesforce. All 29 lines get built into PricebookEntry in V1, giving reps full self-service to quote a complete exhibit if they want to; the stage gate still only requires the anchor.
